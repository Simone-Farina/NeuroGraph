1. HIGH-LEVEL PROJECT IDEA (PRODUCT VISION)
============================================================================================
NeuroGraph is a generative mastery-learning environment.
Its core promise is to convert open-ended exploration into durable understanding.
It avoids two failure modes at once:
- Rigid top-down curriculum paths that suppress curiosity.
- Entropic note-taking that accumulates text without mastery structure.

NeuroGraph sits between those extremes:
- Conversation is the discovery engine.
- Crystallization creates durable conceptual units.
- Retention discipline keeps those units retrievable over time.

Core loop: Organic Discovery -> Crystallized Knowledge -> Rigorous Retention.

Gas-to-crystal cognitive loop:
- Gas phase (exploration through chat):
  - The user thinks out loud with AI.
  - Questions, wrong turns, and reformulations are treated as useful signals.
  - The system optimizes for conceptual movement, not immediate correctness.
- Phase transition (crystallization):
  - A background process monitors depth and conceptual density.
  - When understanding becomes structurally meaningful, a node is created.
  - This is selective to avoid noise and conservative against duplicates.
- Solid phase (graph + retention):
  - The concept joins a persistent knowledge graph.
  - Memory decay is treated as a product reality, not a user failure.
  - Review cycles maintain retrieval fitness and long-range coherence.

Pedagogical principles:
- Atomicity:
  - Each node should represent one indivisible concept.
  - Atomic granularity improves diagnostics, remediation, and transfer.
- Strict prerequisite structure:
  - The graph behaves like a directed dependency structure.
  - Advanced understanding depends on mastered foundations.
- Non-interference:
  - Explanation and verification are separate phases.
  - Verification is intentionally gating to reduce illusion of competence.
- FIRe (fractional implicit repetition):
  - Success on higher-level tasks can reinforce prerequisite concepts indirectly.
  - Intended to produce trickle-down reinforcement across dependencies.
  - This principle is part of the product model even where implementation is pending.

Concept model domains:
- Node as concept:
  - Captures a concise definition and durable core insight.
  - Represents what is learned, not merely what was said.
- Edge as learning relationship:
  - Encodes prerequisite, enabling, or related-concept links.
  - Supports both navigation and pedagogical influence.
- User state over concept:
  - Tracks personal mastery and memory health for each concept.
  - Keeps conceptual truth global while mastery remains individual.

Graph quality thesis:
- Growth quality matters more than growth count.
- Node creation should follow demonstrated conceptual depth.
- The graph should become denser in value, not merely larger in size.

Anticipatory structure:
- Ghost-node style projections are intended to suggest likely next concepts.
- These are navigational affordances, not commitments.
- They support curiosity while preserving learner autonomy.

Retention stance:
- Passive exposure is insufficient for durable knowledge.
- Retrieval practice is required to keep the graph operational.
- Stability is continuous maintenance, not a one-time event.

Dependency-health stance:
- Downstream learning quality depends on foundation quality.
- If prerequisite confidence falls, progress pressure should favor repair first.
- Product success is measured by long-range coherence, not short-term completion.

Intended user journey:
- Explore -> crystallize -> practice -> re-explore.
- Repeated cycles increase graph fidelity and user mastery.
- Over time, the learner accumulates a living map of validated understanding.

Differentiator:
- Graph-first is not a secondary visualization.
- The graph is the primary operational interface for knowledge and progress.

Divergence note (factual deltas only):
- Original architecture drafts described Neo4j + Pinecone + relational storage.
- Current implementation uses Supabase Postgres + pgvector as a consolidated stack.
- Original frontend draft referenced Next.js 15; implementation is currently Next.js 14.2.
- Original retention framing centered on SM-2; implementation now uses FSRS via ts-fsrs.
- Ghost Nodes and FIRe remain intended but are not yet implemented.

Scope discipline reminder:
- Vision defines destination and learning philosophy.
- State and roadmap sections define what is actually live now.
- Keep these layers separate to avoid planning drift.

#!/usr/bin/env node

import nextEnv from '@next/env';

const { loadEnvConfig } = nextEnv;
loadEnvConfig(process.cwd());

const requiredAlways = [
  'NEXT_PUBLIC_SUPABASE_URL',
  'NEXT_PUBLIC_SUPABASE_ANON_KEY',
  'OPENAI_API_KEY',
];

const validProviders = new Set(['openai', 'anthropic', 'google', 'mock']);
const providerRaw = process.env.AI_PROVIDER ?? 'openai';
const provider = providerRaw.toLowerCase();

const missing = [];

for (const key of requiredAlways) {
  if (!process.env[key]) {
    missing.push(key);
  }
}

if (!validProviders.has(provider)) {
  console.error(`Invalid AI_PROVIDER: "${providerRaw}".`);
  console.error('Valid values: openai, anthropic, google, mock');
  process.exit(1);
}

if (provider === 'anthropic' && !process.env.ANTHROPIC_API_KEY) {
  missing.push('ANTHROPIC_API_KEY');
}

if (provider === 'google' && !process.env.GOOGLE_API_KEY) {
  missing.push('GOOGLE_API_KEY');
}

if (!process.env.NEXT_PUBLIC_APP_URL) {
  console.warn('Warning: NEXT_PUBLIC_APP_URL is not set.');
  console.warn('Set it to your Vercel URL for production links and callbacks.');
}

if (provider === 'mock') {
  console.warn('Warning: AI_PROVIDER=mock is intended for local test environments.');
}

if (missing.length > 0) {
  console.error('\nDeployment preflight failed. Missing environment variables:');
  for (const key of missing) {
    console.error(`- ${key}`);
  }
  process.exit(1);
}

console.log('Deployment preflight passed. Required environment variables are set.');

## Schema Overview

### Tables

- **conversations**: User chat sessions
- **messages**: Individual messages within conversations
- **crystals**: Knowledge nodes with embeddings and spaced repetition metadata
- **crystal_edges**: Relationships between crystals

### Indexes

- HNSW index on `crystals.embedding` for fast vector similarity search
- B-tree indexes on foreign keys and `next_review_due`

### RLS Policies

All tables have Row Level Security enabled:
- Users can only access their own data
- Policy: `auth.uid() = user_id`

### Functions

- `find_similar_crystals(embedding, user_id, threshold, limit)`: Vector similarity search
- `get_crystal_neighborhood(crystal_id, max_depth)`: Recursive graph traversal (2-hop max)

## Query Helpers

Import from `@/lib/db/queries`:

```typescript
import { crystalQueries, edgeQueries, conversationQueries, messageQueries } from '@/lib/db/queries';

await crystalQueries.create({ ... });
await crystalQueries.getById(id);
await crystalQueries.getDueForReview(userId);
await crystalQueries.findSimilar(embedding, userId);
await crystalQueries.getNeighborhood(crystalId, 2);
```

## Troubleshooting

### pgvector extension not available

If you get `extension "vector" does not exist`:

1. Go to Supabase Dashboard → Database → Extensions
2. Enable "vector" extension
3. Re-run migrations

### RLS blocking queries

If queries return empty results unexpectedly:

1. Check that you're authenticated: `supabase.auth.getUser()`
2. Verify user_id matches: `auth.uid()` should equal the row's `user_id`
3. Temporarily disable RLS for debugging: `ALTER TABLE table_name DISABLE ROW LEVEL SECURITY;` (NOT for production)

# Feature: Crystal-as-Page

## Problem
Crystals need to become durable knowledge artifacts, not just brief summaries. Users need a page-like surface to expand a crystal with examples, personal notes, and links. Without this, long-term use becomes fragmented (notes live elsewhere) and the graph loses value as the primary interface.

## User Story
As a learner, I want each crystal to be a rich markdown page so that I can grow my understanding over time and navigate knowledge through the graph.

## Scope

### IN (must deliver)
- [ ] Crystal detail panel supports editing markdown content (edit mode)
- [ ] Crystal detail panel supports rendering markdown content (preview mode)
- [ ] Markdown is persisted to the existing crystal content field and reloads on refresh
- [ ] Wiki-link style references (`[[Crystal Title]]`) are detected and displayed as outbound links
- [ ] Backlinks list exists (crystals that reference the current crystal)
- [ ] Basic crystal search/filtering exists to jump between pages

### OUT (explicitly excluded)
- [ ] Offline-first sync and conflict resolution
- [ ] Real-time collaboration
- [ ] Full Obsidian plugin ecosystem
- [ ] Multi-pane workspace layout
- [ ] WYSIWYG editor (rich toolbar, complex formatting UI)

## Technical Design

### Data Model Changes
- No new required fields if `crystals.content` already exists.
- Backlinks can be computed dynamically by parsing content, or stored in a dedicated table later if performance requires it.
- If title-based wiki-links are insufficient, add an ID-based reference format in a follow-up (OUT of scope for this spec).

### API Changes
- Ensure there is a supported update path for crystal content (example: `PATCH /api/crystals/[id]` to update `content`).
- If API already exists, extend validation so `content` accepts markdown text.
- Add (or extend) an endpoint to query backlinks efficiently if the UI requires it.

### UI Changes
- Replace the plain content textarea in the crystal detail panel with:
  - Edit mode: markdown editor surface
  - Preview mode: markdown renderer with GFM (tables, lists, code blocks)
- Add links section:
  - Outbound links: parsed `[[...]]` references
  - Backlinks: other crystals referencing current one
- Add a simple search/filter input to jump between crystals/pages.

## Acceptance Criteria
- [ ] Editing a crystal's markdown persists and reloads correctly (no data loss)
- [ ] Markdown renders with GFM features (lists, tables, fenced code blocks)
- [ ] Rendering does not allow unsafe HTML/XSS by default
- [ ] Outbound wiki-links are detected and shown as clickable items
- [ ] Backlinks list updates after saving content in another crystal
- [ ] Search/filter lets me navigate to a crystal page in under 3 interactions

## Guardrails
- Do NOT refactor chat/crystallization flow as part of this feature
- Do NOT add a complex editor framework unless strictly necessary
- Do NOT render raw HTML from markdown without sanitization
- Do NOT introduce background jobs or new infra for backlinks in this iteration
